# Slice 1 — The World Becomes a Tile Grid

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace 36 hand-authored collision rectangles with a 64px tile grid that is the world's single source of truth — and make the navigation A* run on that same grid, so the `stuckTicks` hack can be deleted rather than fixed.

**Architecture:** A `TileMap` (kinds, not sprites) lives in `src/shared/` because the client collides locally during prediction. `isWalkable()` is the codebase's single collision entry point — `resolveTerrain`, `step`, navigation, monsters and skills all funnel through it — so swapping its body from "iterate rectangles" to "look up tiles" converts everything at once, with free continuous movement untouched. The maps are generated from today's rectangles by a script and committed, so the game plays exactly as it does now.

**Tech Stack:** TypeScript, Vitest inside workerd, `tsx` for the generator script (already in `node_modules/.bin`, no new dependency).

**Spec:** `docs/superpowers/specs/2026-07-13-tiny-swords-world-reset-design.md`

## Global Constraints

- **`src/shared/` is platform-free.** It is compiled by BOTH `tsconfig.client.json` and `tsconfig.worker.json`. No DOM, no Workers API, no Node API, no timers, no randomness. A tilemap the client cannot import breaks prediction on the first wall.
- **Nothing about movement changes.** `step()`, `TICK_HZ`, `PLAYER_SPEED = 260`, diagonal normalisation, one-command-per-tick, `MAX_STARVED_TICKS` — all untouched. We are gridding the *world*, not the walking.
- **Every distance constant stays in pixels.** Attack ranges, `REWARD_DISTANCE`, interest radii, `MINIMAP_WORLD_RADIUS`. The grid changes what a *wall* is, not what a *distance* is.
- **`TILE_SIZE = 64`** — Tiny Swords' native tile size.
- **The world stays 4800 × 2700.** The tilemap is 75 × 43 cells; the bottom row extends past the world edge and is simply solid. Do NOT change `WORLD_WIDTH`/`WORLD_HEIGHT` — they feed boundary rects, camera clamping, the world map's aspect ratio and the minimap, and changing them buys nothing.
- Biome's `noNonNullAssertion` is ON. No `!` assertions — narrow properly.
- Player-facing strings live in `src/shared/i18n/` in BOTH `en.ts` and `fr.ts`. This slice should add none.
- `npm run check` (lint + 3 typechecks + workerd suite + UI suite) must be green before every commit.
- **Never run two `npm run check` invocations at once.** The World Durable Object is a process-wide singleton; concurrent runs make `test/mission-2a.test.ts` time out spuriously.

## What this slice deliberately defers

The spec lists these under Slice 1. They are held back because nothing in this slice renders, and doing them now would be churn against code that Slice 2 rewrites anyway:

- **The autotile lookup table.** Tiles are stored as *kinds*; the 16-entry mask→sprite table only matters once something draws them. It arrives with the renderer.
- **The minimap baking from tiles.** `TerrainGeometry.obstacles` survives this slice, so the minimap keeps working unchanged and correctly. It moves to tiles alongside the terrain renderer.
- **Tiled import/export.** The maps are generated from today's rectangles by a script. Hand-authoring in Tiled only becomes meaningful once you can *see* what you painted.

**At the end of this slice the game looks exactly as it does today.** Collision, navigation and the world's source of truth have moved onto a grid; not one pixel changes on screen. That is the point: if anything looks different, something is wrong.

```ts
/**
 * The world's collision truth, as tiles.
 *
 * Platform-free on purpose: the server runs this to decide where players actually are, and the
 * browser runs the identical code to predict its own square (`net.ts` and `prediction.ts` both
 * collide locally). A tilemap only one side could see would desync on the first wall.
 *
 * This module knows nothing about sprites. It stores what a cell *is*, not what it looks like.
 */
import type { Vec2 } from "./simulation.js";

/** Tiny Swords' native tile size. The art is drawn at this scale; using any other would resample it. */
export const TILE_SIZE = 64;

export type TileKind = "grass" | "plateau" | "water" | "bridge";

/** Row-major, `cols * rows` entries. */
export interface TileMap {
  cols: number;
  rows: number;
  kinds: readonly TileKind[];
}

/** Water is the only barrier; a bridge is the sanctioned way across it. */
export function isSolidKind(kind: TileKind): boolean {
  return kind === "water";
}

/** Outside the map is water: it is a wall, and it needs no special case anywhere else. */
export function kindAt(map: TileMap, col: number, row: number): TileKind {
  if (col < 0 || row < 0 || col >= map.cols || row >= map.rows) return "water";
  return map.kinds[row * map.cols + col] ?? "water";
}

export function kindAtPoint(map: TileMap, x: number, y: number): TileKind {
  return kindAt(map, Math.floor(x / TILE_SIZE), Math.floor(y / TILE_SIZE));
}

/**
 * A body is a box, not a point. Every cell the box touches must be walkable, or a player standing
 * with one shoulder in the water would be allowed to stand there.
 */
export function isWalkableBox(map: TileMap, position: Vec2, size: number): boolean {
  const left = Math.floor(position.x / TILE_SIZE);
  const top = Math.floor(position.y / TILE_SIZE);
  // The box's far edge is exclusive: a body exactly on a cell boundary does not touch the next cell.
  const right = Math.floor((position.x + size - 1) / TILE_SIZE);
  const bottom = Math.floor((position.y + size - 1) / TILE_SIZE);
  for (let row = top; row <= bottom; row++) {
    for (let col = left; col <= right; col++) {
      if (isSolidKind(kindAt(map, col, row))) return false;
    }
  }
  return true;
}
```

```ts
/**
 * Rasterises today's collision rectangles into tile data, once, so the tilemap can become the
 * source of truth without redesigning the world in the same commit.
 *
 * Run with: npm run map:build
 */
import { writeFileSync } from "node:fs";
import { OBSTACLES, type Rect, SAFE_ZONE } from "../src/shared/game.js";
import { type TileKind, TILE_SIZE } from "../src/shared/tilemap.js";
import { VERDANT_REACH_BOUNDS, type WorldBounds } from "../src/shared/simulation.js";
import { TEST_ZONE_TERRAIN } from "../src/shared/zones.js";

/**
 * How much of a cell must be blocked before the cell is a wall.
 *
 * Not "any overlap" (walls would fatten by up to 63px and swallow spawn points that today sit a
 * few pixels from a building), and not "entirely covered" (a 120px-thick wall could straddle cell
 * boundaries and vanish). Half is the rule that preserves both walls and the gaps between them.
 */
const SOLID_COVERAGE = 0.5;

function coverage(rects: readonly Rect[], col: number, row: number): number {
  const x0 = col * TILE_SIZE;
  const y0 = row * TILE_SIZE;
  let covered = 0;
  // Sample on a 8x8 sub-grid: exact enough at 64px, and immune to overlapping rects being
  // double-counted, which a naive area sum would get wrong.
  const STEPS = 8;
  const step = TILE_SIZE / STEPS;
  for (let sy = 0; sy < STEPS; sy++) {
    for (let sx = 0; sx < STEPS; sx++) {
      const px = x0 + (sx + 0.5) * step;
      const py = y0 + (sy + 0.5) * step;
      const hit = rects.some(
        (r) => px >= r.x && px < r.x + r.width && py >= r.y && py < r.y + r.height,
      );
      if (hit) covered++;
    }
  }
  return covered / (STEPS * STEPS);
}

function rasterise(bounds: WorldBounds, obstacles: readonly Rect[]): {
  cols: number;
  rows: number;
  kinds: TileKind[];
} {
  const cols = Math.ceil(bounds.width / TILE_SIZE);
  const rows = Math.ceil(bounds.height / TILE_SIZE);
  const kinds: TileKind[] = [];
  for (let row = 0; row < rows; row++) {
    for (let col = 0; col < cols; col++) {
      kinds.push(coverage(obstacles, col, row) >= SOLID_COVERAGE ? "water" : "grass");
    }
  }
  return { cols, rows, kinds };
}

function emit(name: string, constant: string, data: ReturnType<typeof rasterise>): string {
  // One character per cell, one line per row: a 75-wide map stays readable and diffable in git,
  // and a hand edit in Tiled later shows up as a legible change rather than a wall of numbers.
  const CHAR: Record<TileKind, string> = { grass: ".", plateau: "^", water: "#", bridge: "=" };
  const rowsText = [];
  for (let row = 0; row < data.rows; row++) {
    const line = data.kinds
      .slice(row * data.cols, (row + 1) * data.cols)
      .map((k) => CHAR[k])
      .join("");
    rowsText.push(`  "${line}",`);
  }
  return `// GENERATED by scripts/build-map.ts — do not edit by hand. Run: npm run map:build
import { decodeTileMap } from "../tilemap-codec.js";
import type { TileMap } from "../tilemap.js";

const ROWS = [
${rowsText.join("\n")}
];

/** ${name} */
export const ${constant}: TileMap = decodeTileMap(ROWS);
`;
}

const verdant = rasterise(VERDANT_REACH_BOUNDS, OBSTACLES);
writeFileSync(
  "src/shared/zones/verdant-reach-tiles.ts",
  emit("Verdant Reach", "VERDANT_REACH_TILES", verdant),
);

const test = rasterise(TEST_ZONE_TERRAIN, TEST_ZONE_TERRAIN.obstacles);
writeFileSync(
  "src/shared/zones/mmo-test-zone-tiles.ts",
  emit("Crossing Annex", "MMO_TEST_ZONE_TILES", test),
);

console.log(
  `verdant-reach ${verdant.cols}x${verdant.rows}, mmo-test-zone ${test.cols}x${test.rows}`,
);
console.log(`safe zone spans x ${SAFE_ZONE.x}..${SAFE_ZONE.x + SAFE_ZONE.width}`);
```

Create `src/shared/tilemap-codec.ts`:

```ts
/** Decodes the one-char-per-cell rows that `scripts/build-map.ts` emits. Kept apart from
 *  tilemap.ts so the pure model has no knowledge of how it happens to be stored. */
import type { TileKind, TileMap } from "./tilemap.js";

const KIND: Record<string, TileKind> = {
  ".": "grass",
  "^": "plateau",
  "#": "water",
  "=": "bridge",
};

export function decodeTileMap(rows: readonly string[]): TileMap {
  const first = rows[0];
  if (first === undefined) throw new Error("tile map has no rows");
  const cols = first.length;
  const kinds: TileKind[] = [];
  for (const row of rows) {
    if (row.length !== cols) throw new Error(`ragged tile map: expected ${cols} columns`);
    for (const char of row) {
      const kind = KIND[char];
      if (kind === undefined) throw new Error(`unknown tile character: ${char}`);
      kinds.push(kind);
    }
  }
  return { cols, rows: rows.length, kinds };
}
```

- [ ] **Step 1: Understand what you are deleting**

Read `src/server/world/navigation-system.ts:65-77` (`createNavigationGrid`) and `src/server/world/monster-system.ts:253-263` (the stuck heuristic).

The grid samples **one point** per 48px cell and asks whether a 32px player fits there. The cell is bigger than the sample, so A* routes monsters through corners the continuous collision then refuses; twenty ticks of no progress later, `stuckTicks` throws the path away. The heuristic is not a safety net — it is the cost of the navigation grid and the collision model disagreeing about what the world is.

Once the navigation grid **is** the tilemap, that disagreement is structurally impossible, and the heuristic has nothing left to catch.

```ts
   *  monsters through walls the simulation then refuses — which is what `stuckTicks` used to hide. */

```

In `src/server/world/navigation-system.ts`, rewrite `createNavigationGrid` so a cell is walkable exactly when the tilemap says so — no sampling, no approximation:

```ts
export function createNavigationGrid(
  terrain: TerrainGeometry,
  definition: ZoneNavigationDefinition,
): NavigationGrid {
  const columns = terrain.tiles.cols;
  const rows = terrain.tiles.rows;
  const walkable = new Uint8Array(columns * rows);
  for (let node = 0; node < walkable.length; node++) {
    const col = node % columns;
    const row = Math.floor(node / columns);
    walkable[node] = isSolidKind(kindAt(terrain.tiles, col, row)) ? 0 : 1;
  }
  return { cellSize: definition.cellSize, columns, rows, walkable, terrain };
}
```

Import `isSolidKind` and `kindAt` from `../../shared/tilemap.js`.

**Careful:** a monster is 32px and a cell is 64px, so a cell being walkable does not guarantee a 32px body fits *anywhere* in it — but it does guarantee the body fits *somewhere* in it, and `pointForNode` already centres the body in the cell. Check `pointForNode` still centres correctly with `cellSize = 64` and `PLAYER_SIZE = 32`; the existing arithmetic should hold.
